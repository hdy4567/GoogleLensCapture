# 📋 Google Lens Capture 프로젝트 회고록

프로그램 개발 및 복구 과정에서 발생했던 연쇄적인 실패 원인들을 정밀 분석하여 기록합니다.

---

### 1. 🚨 구글 서버 세션 가로채기 (Session Hijacking)
*   **원인**: `HttpClient`가 업로드 후 리다이렉트 주소를 자동으로 따라감 (`AllowAutoRedirect = true`).
*   **상세**: 프로그램이 결과 페이지에 먼저 '노크'를 하는 바람에, 브라우저가 정작 그 주소로 들어갔을 때는 "이미 사용된 또는 유효하지 않은 세션"으로 인식되어 **'이미지 찾을 수 없음'** 에러 발생.
*   **교훈**: 서버 직접 업로드 시 리다이렉트는 프로그램이 아닌 **브라우저가 첫 번째 소유자**가 되도록 설계해야 함.

### 2. 🧩 라이브러리 참조 모호성 (Ambiguity)
*   **원인**: WPF(`System.Windows`)와 WinForms(`System.Windows.Forms`)의 `Clipboard`, `MessageBox` 클래스 명칭 중복.
*   **상세**: `using` 문이 섞이면서 컴파일러가 어떤 기능을 써야 할지 몰라 빌드 에러 발생. 특히 브라우저 붙여넣기에는 WinForms용 클립보드가 압도적으로 안정적이었으나, 이를 혼용하며 오류 발생.
*   **교훈**: 하이브리드 라이브러리 사용 시 반드시 **Namespace를 명시적으로 지정**하여 충돌을 원천 차단해야 함.

### 3. 🛡️ 클립보드 데이터 전송 순서 미스 (Race Condition)
*   **원인**: 브라우저 실행 후 클립보드에 데이터를 넣으려 했던 시도.
*   **상세**: 브라우저가 포커스를 잡는 타이밍과 프로그램이 `Ctrl+V`를 입력하는 타이밍이 어긋나 붙여넣기 실패.
*   **교훈**: **지능적 순서 변경**을 통해 브라우저가 뜨기 1초 전부터 클립보드에 이미지를 심어두는 '선제적 준비'가 필수임.

### 4. 🔗 필수 파라미터 누락 (Missing Parameters)
*   **원인**: `re=df`, `st`, `ep=gsb` 등 구글 서버가 요구하는 정밀 파라미터 미준수.
*   **상세**: 단순히 이미지만 던지면 구글 서버는 결과 페이지로 보내주지 않고 업로드 성공 코드(200 OK)만 주고 멈춤.
*   **교훈**: 구글 서비스의 최신 엔드포인트 규격(V3)을 **정밀하게 타격**하는 파라미터 조합이 성능의 핵심임.

### 5. 🔒 프로세스 잠금 (File Locking)
*   **원인**: 프로그램이 백그라운드에서 완전히 종료되지 않은 상태로 재작업 시도.
*   **상세**: 이전 실행 인스턴스가 `exe` 파일을 점유하고 있어 새로운 빌드가 실패하거나, 핫키 등록 중복으로 런타임 에러 발생.
*   **교훈**: 작업 전 반드시 **기존 프로세스 완전 종료(taskkill)**를 전제로 한 빌드 및 배포 루틴이 필요함.

---
**최종 결론**: 
단순한 '기능 구현'보다 **"구글 서버의 세션 정책"**과 **"윈도우 시스템 자원(클립보드/핫키)의 정밀한 타이밍 제어"**가 이 프로젝트의 Peak Performance를 결정짓는 핵심 요소였음.
